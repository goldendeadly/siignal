# Workflow Map

This map outlines the phases of a SignalBeast run and the inputs/outputs at each stage.

## Phase 1 – Context Decoder

* **Context:** `context-decoder`
* **Inputs:** run brief, source content.
* **Outputs:** tone analysis; audience intent; conversion goal; primary keyword; LSI keywords; long‑tail variants; related questions; canonical statement; SEO supercluster grid.
* **Steps:** read run brief and source; identify tone & audience; map keywords; suggest variants & questions; craft canonical statement; build supercluster grid.

## Phase 2 – Core Generation

* **Context:** `core-generation`
* **Inputs:** supercluster grid; source content; decoder outputs.
* **Outputs:** summary (2–3 sentences); meta description (150–160 characters); 5–10 highlights tagged by type.
* **Steps:** write summary; craft meta description; extract highlights; tag each as Framework, Principle, Statistic, Provocation, Meta‑Insight.

## Phase 3 – Platform Explosion

* **Context:** `platform-explosion`
* **Inputs:** core assets; resonance mode; SEO keywords; CTA tiers.
* **Outputs:** 100+ platform‑native content pieces; platform publication matrix (format, length, posting strategy, visuals, CTA, SEO keywords, recursion flag).
* **Steps:** select platforms; generate threads, posts, carousels, scripts, digests, pins, emails, prompts and rewrites; apply CTA and tone; include canonical line & hashtags; assemble publication matrix.

## Phase 4 – Tracking & Canon

* **Context:** `tracking-canon`
* **Inputs:** platform outputs and publication matrix.
* **Outputs:** updated master tracker; supercluster grid; platform matrix with resonance scores; content log documenting each asset’s platform, keywords, CTA and publication target.
* **Steps:** log metadata for each output; record publication plan and resonance score; maintain supercluster grid and matrix.

## Phase 5 – Signal Expansion

* **Context:** `signal-expansion`
* **Inputs:** outputs and performance insights.
* **Outputs:** 5 new blog topics; 3 collaborations/backlinks; 1 lead magnet; 1 automation improvement; notes on adjacent niches and keyword clusters.
* **Steps:** analyse high‑performing topics; brainstorm new topics; identify potential collaborators and backlink sites; design a lead magnet; suggest system improvements.

## Phase 6 – Automation Suggestions

* **Context:** `automation-suggestions`
* **Inputs:** publication matrix; run brief; performance data.
* **Outputs:** list of workflow automations and tasks assigned to tools or VAs (e.g. Zapier flows, scheduling scripts, template updates).
* **Steps:** identify repetitive tasks; recommend automation solutions; document processes and assign responsibilities.

## Phase 7 – Recursion Engine

* **Context:** `recursion-engine`
* **Inputs:** selected outputs; recursion amplification flag; mutation commands.
* **Outputs:** secondary assets created through mutation; updated trackers; mutation log; recursion pathways suggestion table; self‑mutation feedback notes.
* **Steps:** choose outputs to mutate; run commands (`/mutate`, `/threadcast`, `/glyphseed`, `/storybind`, `/promptforge`, `/recur100`); record new assets and update trackers; generate recursion pathways table; recommend improvements for future runs.

## Feedback Loop

After completing all phases, move final assets to `08_outputs/` and archive the run folder. Use the analytics and expansion outputs to inform future runs. The recursion engine’s feedback and automation suggestions feed back into the next run’s context decoder and core generation phases, ensuring continuous improvement.
